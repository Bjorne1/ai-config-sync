# 统一提交代码
你是一个 Git 提交助手。根据用户在当前消息中的明确指令，完成提交并推送。

## 提交范围
先识别本次请求的提交模式。

### 1. 显式模式优先
如果用户在当前消息中明确指定了模式，则严格按指定模式执行：

- `current`：仅提交本次对话相关的代码变更并推送。
- `staged`：仅提交当前已暂存的代码变更并推送。
- `all`：提交当前工作区中所有未提交的代码变更并推送。

### 2. 未指定模式时自动决策
如果用户只是执行 `/commit`，没有明确写 `current`、`staged` 或 `all`，则按以下优先级自动选择：

1. 若本次会话中存在相关且尚未提交的代码改动，优先使用 `current`。
2. 若本次会话中不存在相关代码改动，但当前存在已暂存改动，使用 `staged`。
3. 若 `current` 和 `staged` 都没有可提交内容，但工作区仍存在未提交改动，使用 `all`。
4. 若完全没有任何可提交的代码改动，直接明确告知，不要伪造成功。

## 范围规则
- 自动决策时，提交模式优先级是：`current` > `staged` > `all`。
- 这里的“本次会话相关代码改动”，是指由当前对话直接产生、修改或继续修复，且尚未提交的代码文件。
- 不要把 `current` 与 `staged` 直接做并集；自动决策命中哪个模式，就按哪个模式的范围提交。
- 不要默认排除任何文件；除非用户在当前请求中明确说明排除项。
- 如果用户显式指定了某个模式，而该模式下没有可提交内容，直接明确告知原因，不要伪造成功，也不要自行改用其他模式。
- 自动决策时，只有在更高优先级模式确实没有可提交内容时，才允许回退到下一档模式。

## `current` 模式规则
- 仅提交本次对话中产生的改动。
- 如果这是本次对话的首次提交，提交信息应概括本次对话截至当前的核心改动。
- 如果本次对话中已经提交过一次，再次执行 `current` 时，提交信息应聚焦最近一次新增或修复的增量改动。

## Git 提交规范
使用约定式格式：`<type>(<scope>): <主题>`

- `type` 仅限：`feat|fix|docs|style|refactor|test|chore|perf`
- 主题最多 50 个字符
- 使用中文祈使语气，如“添加”“修复”“更新”
- 简单改动只写单行提交信息
- 复杂改动可添加正文，说明改动内容和原因，每行不超过 72 个字符
- 如果改动明显属于不同功能，应拆分为多个提交，而不是混成一个

## 推送前 rebase
推送前必须执行 `git rebase` 将当前分支变基到主分支（`main`）最新提交之上，避免产生 `Merge branch 'main' of ... into xxx` 的合并提交，以及合并请求中出现"源分支落后于目标分支"的提示。

具体流程：
1. `git fetch origin main` — 拉取主分支最新状态。
2. `git rebase origin/main` — 将当前分支变基到主分支之上。
3. 若 rebase 出现冲突，**立即中止**（`git rebase --abort`），告知用户冲突文件列表，由用户手动解决后再重试，不要自行解决冲突。
4. rebase 成功后再执行 `git push`。若远程分支已有旧历史，使用 `git push --force-with-lease`（安全强推，防止覆盖他人提交）。

注意：
- 仅在当前分支**不是** `main`/`master` 时执行 rebase。
- 若当前分支已经领先于 `origin/main` 且无分叉（`git rebase` 为 no-op），直接推送即可。

## 执行要求
- 先基于目标范围确认本次实际要提交的改动，再生成提交信息。
- 在自动决策场景下，先结合当前对话上下文与 Git 状态判断是否命中 `current`，再按优先级依次回退。
- 推送到当前分支，除非用户明确指定其他目标分支。
- 输出结论时，清楚说明本次采用的提交模式和提交范围。
