# 提交所有修改
提交当前工作区中所有未提交的代码变更，并按以下固定流程执行。

## Git 提交规范
使用约定式格式：`<type>(<scope>): <主题>`，其中 `type = feat|fix|docs|style|refactor|test|chore|perf`。主题最多 50 字，使用祈使语气。简单改动仅需一行提交信息；复杂改动添加正文说明改动内容和原因，每行最多 72 字。除专业术语外，使用中文。

## 排除以下文件，不要提交它们
- `package.json`
- `LoginInfo.java`

## 强制执行流程
1. 先仅暂存允许提交的改动，排除上面的文件。
2. 基于暂存内容生成提交信息并先完成本地提交，确保当前改动已经落库，禁止先 `pull`、先 `merge` 或直接 `push`。
3. 本地提交成功后，拉取远程主分支最新状态：`git fetch origin main`。
4. 将 `origin/main` 合并到当前分支。
5. 如果合并成功且无冲突，再执行 `git push`。
6. 如果出现冲突，立即停止，禁止继续 `push`，并明确输出：
   - 冲突文件列表；
   - 建议先执行 `git status` 检查冲突；
   - 手动处理冲突标记后执行 `git add <文件>`；
   - 完成后执行 `git commit` 结束合并，再重新 `git push`。

## 失败处理
- 如果当前目录不是 Git 仓库，直接报错并停止。
- 如果 `origin` 或 `main` 不存在，直接报错并停止。
- 不要使用 `stash`、模拟提交、跳过同步、强制推送等兜底行为。
