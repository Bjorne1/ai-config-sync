# 提交当前对话代码变更
仅提交本次对话中修改的代码，并按以下固定流程执行。

## 逻辑判断
回顾本次对话上下文：
1. 如果这是本次对话的首次提交，提交信息应覆盖本次对话实现的核心功能。
2. 如果此前已经生成过提交命令，本次提交信息只聚焦最近一次修改或修复。

## Git 提交规范
使用约定式格式：`<type>(<scope>): <主题>`，其中 `type = feat|fix|docs|style|refactor|test|chore|perf`。主题最多 50 字，使用祈使语气。简单改动仅需一行提交信息；复杂改动添加正文说明改动内容和原因，每行最多 72 字。除专业术语外，使用中文。

## 排除文件
- `package.json`
- `LoginInfo.java`

## 强制执行流程
1. 只暂存本次对话实际修改且允许提交的文件。
2. 基于这些暂存内容先完成本地提交，确保当前修改不会丢失，禁止先同步远程再提交。
3. 本地提交成功后执行 `git fetch origin main`。
4. 将 `origin/main` 合并到当前分支。
5. 如果没有冲突，执行 `git push`。
6. 如果存在冲突，立即停止，不得继续 `push`，并输出：
   - 具体冲突文件；
   - 建议使用 `git status` 查看状态；
   - 手动解决冲突标记后执行 `git add <文件>`；
   - 再执行 `git commit` 完成合并，随后重新推送。

## 失败处理
- 当前目录不是 Git 仓库时直接停止。
- 无法获取 `origin/main` 时直接停止。
- 不要自动 `stash`、跳过合并、强推或编造成功结果。
